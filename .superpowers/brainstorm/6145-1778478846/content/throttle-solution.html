foxglove_bridge Frequency Throttle — Three-Layer Solution

Layer 1: ROS topic_tools throttle · Layer 2: Protobuf wire encoding · Layer 3: Browser rAF render cap

Problem Statement

DEMO-1 (6/15)

2 vessels (own-ship + 1 target)
50Hz nav state + 2Hz tracks
~53 msg/s → JSON ~50 KB/s
No problem, modern browser handles easily

DEMO-2 (7/31)

50+ vessels (own-ship + 50 targets)
50Hz nav state + 2Hz × 50 tracks
~150 msg/s → JSON ~500 KB/s
HTML DOM + WebSocket parse + MapLibre render = bottleneck

Layer 1: ROS2 topic_tools throttle (server-side republish)

Phase 1 (DEMO-1): Add throttle node to launch file
# Install (once)
sudo apt install ros-humble-topic-tools

# Launch: throttle nav data 50Hz → 10Hz
ros2 run topic_tools throttle messages \
  /nav/filtered_state 10 /nav/filtered_state_viz

# Launch: throttle tracks 2Hz → 2Hz (pass-through, already low)
ros2 run topic_tools throttle messages \
  /world_model/tracks 2 /world_model/tracks_viz

# foxglove_bridge subscribes to *_viz throttled topics
ros2 launch foxglove_bridge foxglove_bridge_launch.xml \
  port:=8765 \
  topic_whitelist:='[".*_viz", "/l3/.*"]'

Layer 2: Protobuf Binary Encoding (DEMO-2 switch)

JSON (DEMO-1)

✅ Human-readable, easy debug
✅ Zero config — foxglove_bridge default
✅ Browser WebSocket text frames
❌ 3-10x larger than Protobuf
32 KB/s → fine for DEMO-1

Protobuf (DEMO-2)

✅ 3-10x smaller payload
✅ Binary WebSocket frames (faster parse)
✅ Schema auto-advertised by bridge
⚠️ Requires .proto defs for custom messages
32 KB/s → ~5 KB/s with 50 vessels

🟡 Medium confidence — Protobuf gain varies by message structure. foxglove_bridge supports both modes natively. Source: Foxglove blog

Layer 3: Browser-Side Throttle (always active)

React hook: useThrottledTopic(ws, topic, maxFps)
// Custom hook: throttled WebSocket → React state
function useThrottledTopic(ws, topic, maxFps=30) {
  const [data, setData] = useState(null);
  const lastUpdate = useRef(0);
  const minInterval = 1000 / maxFps;

  useEffect(() => {
    ws.on('message', (msg) => {
      const now = performance.now();
      if (now - lastUpdate.current >= minInterval) {
        lastUpdate.current = now;
        setData(msg); // triggers MapLibre setData()
      }
    });
  }, [ws, topic, minInterval]);

  return data;
}

Decision: Start with topic_tools throttle + JSON, upgrade to Protobuf at DEMO-2

Pros

  • No custom code — official ROS2 package
  • Throttle rate per-topic, adjustable at launch
  • JSON for DEMO-1 dev speed, Protobuf for DEMO-2 performance
  • Zero foxglove_bridge modification needed

Cons

  • Extra ROS2 node in launch file (minimal overhead)
  • Protobuf requires .proto message definitions for custom IDL types
  • roslibjs-foxglove does NOT support Protobuf mode — need @foxglove/ws-protocol for DEMO-2

🟡 Medium — Protobuf upgrade path clear but roslibjs-foxglove fork needed for binary mode. Acceptable for DEMO-2 timeline.
