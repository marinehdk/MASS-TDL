## 1. 概述与职责定义

### 1.1 模块定位

Hardware Override Arbiter（硬线仲裁器）是 v2.0 防御性架构的 Z 轴底层安全基座。它是一个纯硬件模块——由继电器、信号选择器和模拟电路组成，不包含任何软件或可编程逻辑。当桥楼值班船员（OOW）物理操作车钟或舵轮时，硬线仲裁器以 P0 最高优先级切断所有 AI 信号通路，将执行机构（舵机、主机、侧推器）的控制权直接移交给人工输入。

### 1.2 核心设计原则

- **零软件参与：** 切换动作完全由物理电路完成。即使所有计算机崩溃、操作系统死机、ROS2 彻底停止，人工接管仍然有效。
- **零延迟切换：** 从人工操作到信号切换的延迟 < 10ms（继电器动作时间）。
- **信号选择而非信号处理：** 仲裁器不修改信号内容——它只是一个"开关"，选择 AI 信号还是人工信号通过。

## 2. 硬件架构设计

### 2.1 信号路径

```
AI 控制信号（来自 L5）─────────┐
                                │    ┌──────────────┐
                                ├───→│  信号选择器    │───→ 舵机 ECU
人工舵轮信号 ──────────────────→│    │ (继电器组)    │───→ 主机 ECU
                                │    │              │───→ 侧推 ECU
人工车钟信号 ──────────────────→│    └──────┬───────┘
                                │           │
                                │    ┌──────┴───────┐
人工接管检测 ──────────────────→│    │  优先级逻辑    │
(舵轮力矩传感器/车钟位置开关)     │    │ (硬件电路)    │
                                │    └──────────────┘
                                │
通知信号 ←─────────────────────┘ → 到软件系统（只读）
```

### 2.2 核心器件

| 器件 | 型号/规格 | 数量 | 说明 |
|------|---------|------|------|
| 信号继电器 | 工业级双刀双掷 (DPDT) | 3 组 | 舵角、转速、侧推各一组 |
| 接管检测开关 | 舵轮力矩传感器 + 车钟限位开关 | 2 | 检测人工操作 |
| 速率限制器 | 模拟 RC 滤波电路 | 3 组 | 防止切换瞬态冲击 |
| 状态指示 | LED + 蜂鸣器 | 1 组 | 显示当前控制模式 |
| 通知接口 | 干接点输出 | 1 | 通知软件系统（只读） |

### 2.3 继电器选型要求

```
# 继电器必须满足以下要求：
# 1. 海事环境等级（IP56+，防盐雾腐蚀）
# 2. 工作温度范围 -20~60°C
# 3. 动作时间 < 10ms
# 4. 触点寿命 > 10 万次
# 5. 触点容量足以承载控制信号电流
# 6. 失电状态（线圈无电）= 人工模式（安全失效）
```

**关键决策——失电状态选择：**

继电器在线圈断电时（默认状态）应处于哪个位置？

**方案 A：失电 = 人工模式（推荐）**
- 继电器默认（线圈无电）选择人工信号通路
- AI 信号需要继电器通电才能通过
- 安全理由：系统断电时自动切换到人工——这是最安全的失效模式

**方案 B：失电 = AI 模式**
- 不推荐——系统断电时 AI 也没有信号输出，执行机构会停在最后状态

**选择方案 A。**

## 3. 信号路径与优先级

### 3.1 四级优先级（v3.1 扩展）

```
P0（最高）：物理人工操作——继电器级直接切换（舵轮/车钟/接管按钮）
P1：        IJS 独立摇杆——独立 MCU 硬件，不经过 ROS2（v3.1 新增）
P2：        紧急反射弧——软件紧急指令通过 DDS 到 L5
P3（正常）：AI 自主控制——L5 常规输出

P0 覆盖 P1/P2/P3（物理继电器级，无条件）
P1 覆盖 P2/P3（硬件 CAN Bus 级，当主控计算机不可用时自动激活）
P2 覆盖 P3（软件级，通过 ROS2 话题优先级）
P3 为正常模式

降级路径：
  P3（AI 正常）→ 单台计算机故障 → Redundancy Manager 切换 → 保持 P3
                → 双台计算机故障 → 自动降级到 P1（IJS 独立摇杆）
                → IJS 也故障 → 降级到 P0（纯物理人工）

IJS 说明（v3.1，V2 版本实现）：
  独立于 ROS2/DDS/主控计算机的硬件摇杆系统
  拥有自己的 MCU（STM32 级别），通过 CAN Bus 直连推进器控制器
  内置简化推力分配算法（线性映射，非 QP 优化）
  当 Redundancy Manager 的心跳信号消失 3 秒后自动激活
  详见 L5 Mode Switcher 文档的 IJS_EMERGENCY 模式定义
```

### 3.2 信号选择逻辑

```
# 硬件逻辑（用简单的 AND/OR 门或继电器逻辑实现）

IF human_override_detected:
    relay_position = HUMAN    # P0：人工
    notify_software(OVERRIDE_ACTIVE)
ELSE:
    relay_position = AI       # P2/P1 由软件层处理
    notify_software(AI_ACTIVE)
```

### 3.3 人工接管检测

```
人工接管的物理检测方式：

舵轮检测：
  方法 1：舵轮上安装力矩传感器，当施加 > 5 N·m 的力矩时触发
  方法 2：舵轮转动角速度 > 1°/s 且与 AI 指令方向不一致时触发
  方法 3：舵轮旁安装物理接管按钮（最简单可靠）
  
  推荐：方法 3（物理按钮）+ 方法 1 辅助确认

车钟检测：
  方法：车钟手柄的位置编码器，当手柄被人工移动时（位置变化 > 5%）触发
  
侧推检测：
  方法：侧推控制杆的位置开关，任何偏离中位的操作触发
```

## 4. 切换逻辑

### 4.1 Bumpless Transfer 前提——伺服推杆跟随（v3.1 关键升级）

在 AI 模式下切换到人工前，必须确保物理推杆的位置与 AI 当前输出的推力值对齐。否则切换瞬间会产生推力断崖跳变——这在恶劣海况下可能导致船体横倾甚至倾覆。

**致命场景举例：** AI 正在输出 80% 推力抵抗横风保持航向，但驾驶台的物理车钟推杆停在 0%。船长按下接管按钮 → 推力从 80% 断崖跌到 0% → 横风瞬间推动船体 → 船体剧烈横倾 → 可能损坏推进器齿轮箱甚至倾覆。

**解决方案：** 详见第 14 节"Bumpless Transfer 无扰动切换机制"。核心要求——AI 模式下物理推杆由内置伺服电机驱动自动跟随 AI 输出值，船长接管时推杆位置 = AI 当前推力 = 实际推力，实现零跳变。

### 4.2 从 AI 切换到人工（接管）

```
前提条件：伺服推杆已对齐（servo_aligned = true）

动作序列（全部由硬件完成，< 10ms）：
1. 伺服推杆对齐确认（推杆位置与 AI 输出偏差 < 3%）
2. 继电器线圈断电 → 触点切换到人工信号通路
3. AI 信号通路物理断开
4. 人工信号直接连接到执行机构
5. 通知灯亮起（红色 = 人工模式）
6. 蜂鸣器短响一声
7. 干接点信号通知软件系统 (override_active = true)

紧急覆写：
  即使 servo_aligned = false（伺服故障），船长仍然可以强制接管——
  此时启用软件斜坡限幅保护（Ramp Limiter，见第 14.4 节），
  将推力变化率限制在安全范围内。人命安全 > 设备保护 > 平滑性。
```

## 5. 瞬态保护

### 5.1 三层防护体系（v3.1 升级）

瞬态保护从单一的 RC 滤波器升级为三层防护——每一层独立工作，任何一层生效都能防止设备损坏：

```
┌──────────────────────────────────────────────────────────────┐
│ 第一层：物理消除跳变源头（伺服推杆跟随）                        │
│   AI 模式下推杆物理位置 = AI 推力值                            │
│   切换时推杆位置 = 当前推力 → 源头上就没有跳变                  │
│   详见第 14 节                                                │
├──────────────────────────────────────────────────────────────┤
│ 第二层：硬件速率限制（模拟 RC 滤波器）                          │
│   即使第一层失效（伺服故障），RC 滤波器限制信号变化速率           │
│   τ = 0.5~1.0s，防止瞬间阶跃到达执行机构                       │
├──────────────────────────────────────────────────────────────┤
│ 第三层：软件斜坡限幅（Ramp Limiter）                           │
│   即使第一、二层都失效，Actuator Limiter 的速率限幅仍然生效      │
│   舵角变化率 ≤ 5°/s，RPM 变化率 ≤ 100/s                       │
│   这是已有的 v2.0 功能——L5 Actuator Limiter 文档已定义          │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 第二层详细设计：模拟速率限制器

```
在信号选择器输出端串联模拟 RC 低通滤波电路：

R = 10kΩ, C = 100μF → τ = RC = 1s

效果：即使伺服推杆故障导致信号阶跃 65°，输出以 ~65°/s 渐变
    → 约 1 秒内从 +30° 过渡到 -35°
    → 远低于舵机的机械速率限制（5°/s 时需要 13 秒）

建议 τ = 0.5~1.0 秒
```

### 5.3 主机换向保护

主机（螺旋桨）的换向比舵角切换更危险——从全速前进直接倒车可能损坏传动系统。

```
# 主机换向保护电路（硬件实现）：
# 在方向反转时：
# 1. 先将转速降至 0（停机）
# 2. 等待 T_reversal_delay 秒
# 3. 再按新方向启动

T_reversal_delay = 5 秒（需根据主机型号确认）
```

### 5.4 反射弧紧急指令与电力管理的冲突处理（v3.1 新增）

**问题：** Emergency Reflex Arc 设计为"绕过 L3/L4 直插 L5"——但如果连 Power Management 也绕过了，Crash Stop 的瞬间涌流可能拉垮发电机导致全船断电（Blackout），使险情变成完全失控。

```
FUNCTION reflex_with_power_protection(emergency_cmd, power_status):
    
    # 反射弧的紧急指令仍然绕过 L3/L4 的决策校验
    # 但必须经过 Power Management 的功率限幅
    # 区别是：限幅阈值放宽到发电机的短时过载能力
    
    normal_power_limit = power_status.continuous_rating         # 连续额定功率
    emergency_power_limit = power_status.short_term_overload    # 短时过载（110~120%，持续 10~30s）
    
    IF emergency_cmd.total_power_demand > emergency_power_limit:
        # 超过短时过载极限——必须削峰
        # 策略：优先保证转向（舵机功率小），削减主机推力
        scale_factor = emergency_power_limit / emergency_cmd.total_power_demand
        emergency_cmd.rpm *= scale_factor
        # 舵角不削减——转向避碰优先
        
        log_event("reflex_power_limited", {
            demanded: emergency_cmd.total_power_demand,
            allowed: emergency_power_limit,
            scale: scale_factor
        })
    
    RETURN emergency_cmd

# 关键原则：
# 宁可牺牲部分避让幅度（RPM 降低 = 减速更慢）
# 也必须死守全船不断电的生命底线
# 因为断电 = 失去全部控制能力 = 比避让不够更危险
```

## 6. 回切协议

### 6.1 从人工回到 AI 的条件

人工切换回 AI 不能是一个"松手就切回"的瞬间动作——需要安全确认。

```
回切协议（硬件 + 软件配合）：

步骤 1：船员释放舵轮/车钟（回到中立位置或按下"回切"按钮）
步骤 2：通知信号传递给软件系统
步骤 3：软件系统检查：
    - AI 子系统是否正常运行？
    - 传感器是否正常？
    - 是否有未解决的紧急事件？
步骤 4：软件确认可以回切 → 发出回切允许信号
步骤 5：30 秒确认窗口——船员在 30 秒内再次确认（按住回切按钮 3 秒）
步骤 6：继电器通电 → 信号切换回 AI 通路
步骤 7：AI 从当前实际状态开始控制（不是从接管前的状态）
```

### 6.2 AI 重新接管时的初始化

```
FUNCTION ai_resume_after_override():
    
    # AI 重新接管时，必须从当前实际状态开始——不能用接管前的状态
    
    # 1. 重新读取当前本船位置、航向、航速
    current_state = nav_filter.get_latest()
    
    # 2. 重新计算当前在航线上的位置
    los_guidance.reinitialize(current_state, planned_route)
    
    # 3. 清空所有控制器的积分项（防止积分饱和）
    heading_controller.reset_integrator()
    speed_controller.reset_integrator()
    
    # 4. 初始输出 = 当前实际舵角/转速（无缝衔接）
    initial_output = {
        delta: actual_rudder_angle,
        rpm: actual_engine_rpm
    }
    
    # 5. 逐渐过渡到 AI 计算的目标值（5~10 秒过渡期）
    transition_duration = 10    # 秒
```

## 7. 与软件系统的接口

### 7.1 通知信号（硬件 → 软件）

```
# 仲裁器通过干接点输出向软件系统通知当前状态
# 软件通过 GPIO 或数字输入模块读取

信号定义：
  override_active (DI-01):  true = 人工模式, false = AI 模式
  override_type (DI-02):    bit0 = 舵轮, bit1 = 车钟, bit2 = 侧推
  override_request (DI-03): true = 船员按下了回切按钮

# 软件侧处理：
IF override_active:
    # 1. 暂停所有 AI 控制输出
    # 2. 继续运行感知和决策（不停止），只是输出被旁路
    # 3. 记录接管事件到 ASDR
    # 4. 通知岸基
    asdr.log_event("HUMAN_OVERRIDE", {type, timestamp})
    shore_link.alert("HUMAN_OVERRIDE_ACTIVE")
```

### 7.2 软件 → 仲裁器的信号

```
# 软件只有一个输出信号到仲裁器：

ai_ready (DO-01):  true = AI 系统正常可接管, false = AI 故障

# 当 ai_ready = false 时，即使船员按下回切按钮，仲裁器也不切回 AI
# 这防止了 AI 故障时被错误地切回
```

## 8. 故障模式分析

| 故障模式 | 后果 | 检测方式 | 应对措施 |
|---------|------|---------|---------|
| 继电器卡在 AI 位置 | 人工无法接管 | 接管后检测执行机构是否响应人工输入 | 备用硬线直连（紧急旁路开关） |
| 继电器卡在人工位置 | AI 无法恢复 | AI 检测 override_active 但无人工操作 | 手动重置继电器 |
| 接管检测误触发 | AI 被意外切断 | 软件检测 override 状态 + 无人工输入 | 10 秒超时自动恢复 |
| 接管检测失效 | 人工操作无法被检测到 | 定期自检（按下检测按钮测试） | 备用接管按钮 |
| 速率限制器失效 | 切换瞬态冲击 | 执行机构保护（内置限位） | 执行机构自身的电子限位 |
| 通知信号线断开 | 软件不知道已被接管 | 软件检测 AI 输出与实际执行不一致 | 安全侧——软件假设可能被接管 |

## 9. 安装与布线要求

```
# 硬线仲裁器的安装位置：
# - 尽可能靠近执行机构（减少信号线长度）
# - 独立的接线箱（防水 IP56+）
# - 与 AI 计算机分开的电源供应（独立保险丝回路）
# - 布线走独立的线槽（不与 AI 信号线共用线槽）

# 信号线要求：
# - 屏蔽双绞线（防电磁干扰）
# - 标识颜色：红色 = 人工信号，蓝色 = AI 信号，黄色 = 通知信号
# - 连接器：防水航空插头

# 电源要求：
# - 继电器线圈：24V DC（船用标准）
# - 信号选择器：无源（继电器触点只是开关，不需要电源）
# - 接管检测：24V DC（与继电器同源）
```

## 14. Bumpless Transfer 无扰动切换机制（v3.1 关键升级）

### 14.1 问题定义

"Bumpless Transfer"（无扰动切换）是 Kongsberg K-Pos 等工业级 DP 系统的标准功能——在控制权从自动模式转移到人工模式时，执行机构的输出指令没有阶跃跳变。这不是"锦上添花"的舒适性功能，而是安全关键需求。

```
致命场景完整推演：

1. 45m crew boat 正在以 18 节航速横渡，海况 4（Hs=2m），横风 25 节

2. MASS_ADAS 处于 AI AUTO 模式
   - 双喷水推进器输出 78% 推力维持航速
   - 左推进器偏转 8° 补偿横风偏流
   - 横向稳定需要持续 ~15kN 的侧向推力

3. 驾驶台物理操纵杆（车钟/方向推杆）停在中位（0% 推力，0° 偏转）
   因为 AI 自动模式下船长不操控推杆——推杆在什么位置取决于上次人工操作

4. 船长按下"接管"按钮

5. 如果没有 Bumpless Transfer：
   → 推力瞬间从 78% 跌到 0%
   → 偏转从 8° 跳到 0°
   → 18 节的惯性 + 25 节横风 + 零推力
   → 船在 3~5 秒内开始剧烈偏航
   → 78 名乘客感受到强烈横向加速度
   → 推进器齿轮箱承受极端扭矩反转——可能机械损坏
```

### 14.2 伺服电动推杆（Motorized Command Lever）

**核心硬件：** 驾驶台的每个操纵推杆内部集成一台伺服电机——当系统处于 AI 自动模式时，伺服电机驱动推杆物理移动到与 AI 当前输出值一致的位置。

```
推杆跟随控制逻辑（独立的嵌入式微控制器运行，不依赖 ROS2）：

FUNCTION servo_follow_loop():    # 50 Hz 运行
    
    WHILE system_mode == "AI_AUTO" OR system_mode == "REMOTE":
        
        # 读取 AI 当前输出的推力值（通过模拟量/CAN 总线）
        ai_thrust_pct = read_ai_thrust_output()      # 0~100%
        ai_direction_deg = read_ai_direction_output()  # -180~+180°
        
        # 读取推杆当前物理位置（内置电位器）
        lever_thrust_pct = read_lever_position()
        lever_direction_deg = read_lever_angle()
        
        # 伺服电机驱动推杆跟随 AI 值
        thrust_error = ai_thrust_pct - lever_thrust_pct
        direction_error = ai_direction_deg - lever_direction_deg
        
        # 限制伺服移动速率（推杆不能移动太快——船长的手可能放在上面）
        servo_thrust_cmd = clamp(thrust_error × Kp, -SERVO_MAX_RATE, SERVO_MAX_RATE)
        servo_direction_cmd = clamp(direction_error × Kp, -SERVO_DIR_RATE, SERVO_DIR_RATE)
        
        drive_servo(servo_thrust_cmd, servo_direction_cmd)
        
        # 计算对齐状态
        aligned = (abs(thrust_error) < ALIGN_THRESHOLD_PCT 
                   AND abs(direction_error) < ALIGN_THRESHOLD_DEG)
        set_aligned_flag(aligned)
        
        # 对齐指示灯
        IF aligned:
            set_led("green")    # 推杆已对齐——安全接管
        ELSE:
            set_led("amber")    # 推杆未对齐——接管有风险
        
        sleep(20ms)    # 50 Hz

SERVO_MAX_RATE = 20     # %/秒（推力推杆最大移动速率）
SERVO_DIR_RATE = 15     # °/秒（方向推杆最大移动速率）
ALIGN_THRESHOLD_PCT = 3  # 推力对齐阈值：3%
ALIGN_THRESHOLD_DEG = 2  # 方向对齐阈值：2°
Kp = 5.0                 # 比例增益
```

### 14.3 船长手握推杆时的力反馈

当船长把手放在推杆上时（通过推杆上的触摸传感器或力传感器检测），伺服行为改变：

```
FUNCTION servo_with_hand_detection():
    
    IF hand_on_lever_detected():
        
        # 船长手在推杆上——两种可能：
        # A) 他只是扶着（不施加力）→ 伺服继续跟随但力矩降低
        # B) 他在主动操控（施加力）→ 这是接管的前兆
        
        IF hand_force < PASSIVE_HOLD_THRESHOLD:
            # 船长只是扶着——伺服继续跟随但降低力矩（手感更柔和）
            servo_torque = SERVO_TORQUE_REDUCED    # 50% 力矩
        ELSE:
            # 船长在主动推动——进入"力对抗模式"
            # 伺服不再跟随 AI，而是逐渐释放（让船长接管控制）
            # 这实现了"渐进式接管"——不是突然切换，而是平滑过渡
            
            blend = hand_force / MAX_HAND_FORCE    # 0~1
            lever_output = (1 - blend) × ai_value + blend × lever_physical_position
            
            # 当 blend > 0.8 时自动触发完整的 Override 切换
            IF blend > OVERRIDE_BLEND_THRESHOLD:
                trigger_full_override()
    
    ELSE:
        # 船长没碰推杆——正常跟随
        servo_torque = SERVO_TORQUE_NORMAL

PASSIVE_HOLD_THRESHOLD = 5    # N（5 牛顿以下视为被动扶握）
MAX_HAND_FORCE = 30           # N（30 牛顿视为完全主动操控）
OVERRIDE_BLEND_THRESHOLD = 0.8
SERVO_TORQUE_NORMAL = 100     # % 额定力矩
SERVO_TORQUE_REDUCED = 50     # %
```

### 14.4 软件斜坡限幅（Ramp Limiter）——伺服故障时的保底保护

即使伺服推杆故障（电机烧毁、通信中断），切换仍然必须安全。Ramp Limiter 是一个纯软件保护层——运行在 Actuator Limiter 模块中（L5），限制从 Override Arbiter 输出到执行机构的信号变化率。

```
FUNCTION ramp_limit(current_output, new_input, dt):
    
    # 推力变化率限制
    max_thrust_change = THRUST_RAMP_RATE × dt    # %/step
    thrust_change = new_input.thrust - current_output.thrust
    thrust_change = clamp(thrust_change, -max_thrust_change, max_thrust_change)
    limited_thrust = current_output.thrust + thrust_change
    
    # 方向变化率限制
    max_dir_change = DIRECTION_RAMP_RATE × dt    # °/step
    dir_change = normalize_pm180(new_input.direction - current_output.direction)
    dir_change = clamp(dir_change, -max_dir_change, max_dir_change)
    limited_direction = current_output.direction + dir_change
    
    RETURN {thrust: limited_thrust, direction: limited_direction}

# 正常模式——缓慢斜坡（舒适性+设备保护）
THRUST_RAMP_RATE_NORMAL = 10     # %/秒（从 0% 到 100% 需要 10 秒）
DIRECTION_RAMP_RATE_NORMAL = 5   # °/秒

# 紧急模式——快速斜坡（仍有保护但响应更快）
THRUST_RAMP_RATE_EMERGENCY = 30  # %/秒（从 0% 到 100% 需要 3.3 秒）
DIRECTION_RAMP_RATE_EMERGENCY = 15  # °/秒

# 当 Reflex Arc 触发时使用紧急斜坡率
# 当正常接管时使用正常斜坡率
```

### 14.5 接管时序完整流程

```
时序图（正常接管，伺服推杆工作正常）：

T=0s    AI AUTO 模式
        推杆伺服持续跟随 AI 输出
        推杆物理位置 = AI 推力值 = 78%
        对齐指示灯：绿色

T=0s    船长手握推杆（触摸传感器检测到）
        伺服降低力矩（手感变柔）
        
T=0.5s  船长开始主动推动推杆
        进入力对抗模式
        输出 = (1-blend) × AI + blend × 手动
        推力从 78% 平滑过渡

T=1.0s  blend > 0.8 → 自动触发 Override
        继电器切换
        推杆完全由船长控制
        此刻推杆位置 ≈ 75%（接近 AI 的 78%，跳变 < 3%）

T=1.0+  船长正常操控——无跳变、无冲击

总过渡时间：~1 秒
推力跳变：< 3%（远低于危险阈值）
```

```
时序图（紧急接管，伺服推杆故障）：

T=0s    AI AUTO 模式，AI 推力 = 78%
        伺服推杆故障——推杆卡在 0% 位置
        对齐指示灯：红色闪烁（警告）

T=0s    船长强制按下接管按钮
        继电器切换——输入信号从 78% 变为 0%

T=0+    三层防护启动：
        第一层（伺服）：失效
        第二层（RC 滤波）：τ=0.5s，信号以 156%/s 速率渐变
        第三层（软件斜坡）：限制在 30%/s（紧急率）

        实际效果：推力从 78% 以 30%/s 下降
        → 78%→48% (1s) → 18% (2s) → 0% (2.6s)
        → 2.6 秒内推力归零（而非瞬间归零）
        → 船员有时间感知并操作推杆补偿

T=2.6s  推力降到 0%
        船长此时已把推杆推到目标位置
        新的推力值通过斜坡限幅逐步上升
```

### 14.7 伺服推杆失效的安全性分析

```
故障模式分析：

故障 1：伺服电机烧毁
  → 推杆不再跟随 AI（停在当前位置）
  → 对齐指示灯变红
  → 船长接管时有跳变风险
  → RC 滤波器 + 软件斜坡限幅 生效（第二/三层）
  → 安全——降级但可控

故障 2：位置传感器故障
  → 伺服无法准确定位（可能振荡或飘移）
  → 看门狗检测到位置反馈异常 → 关闭伺服
  → 回退到故障 1 的处理

故障 3：CAN 通信中断（无法读取 AI 推力值）
  → 伺服保持最后收到的 AI 值不动
  → 0.5 秒无更新 → 看门狗关闭伺服
  → 回退到故障 1 的处理

故障 4：伺服电机锁死（最危险——推杆无法移动）
  → 船长无法操控推杆！
  → 必须设计机械离合器——船长用力超过阈值时物理脱开电机
  → 推杆变为纯机械操控
  → 对齐不可能——但船长至少能操控

关键设计要求：
  伺服推杆的故障模式必须是"释放"而非"锁死"。
  机械离合器的脱开力设定为 ~50N（大于正常操控力但小于紧急拉扯力）。
```

### 14.8 与全回转推进器的适配

45m Crew Boat 如果使用双全回转推进器（azimuth thrusters），推杆系统更复杂——每台推进器有两个自由度（推力大小 + 偏转角度）。

```
全回转推进器的推杆配置：

选项 A：双摇杆
  左摇杆 → 左推进器（前后=推力，左右=偏转）
  右摇杆 → 右推进器（前后=推力，左右=偏转）
  每个摇杆内置 2 轴伺服电机跟随

选项 B：组合摇杆（Kongsberg cJoy 模式）
  一个三轴摇杆：前后=前进后退，左右=横移，旋转=艏摇
  推力分配由 Thrust Allocator 完成
  摇杆伺服跟随的是 Thrust Allocator 的输入端（力/力矩需求）
  而非输出端（单台推进器指令）

建议：采用选项 B——与 Kongsberg cJoy 设计一致
  优点：船长操控直觉（推前=前进，推右=右移），不需要理解推力分配
  伺服跟随更简单（只跟 3 个值：Fx, Fy, Mz）
```
