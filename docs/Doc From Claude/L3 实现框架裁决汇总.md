## L3 实现框架裁决汇总

### F-01 进程/线程边界 — **2 进程模型** （nlm-plan 4.8/5）

| 进程            | 包含组件                                    | 通信                                     |
| --------------- | ------------------------------------------- | ---------------------------------------- |
| **P1 主决策**   | Stage 1（看）+ Stage 2（判）+ Stage 3（动） | P1 内部：零拷贝 lockfree SPSC queue      |
| **P2 安全监控** | Stage 4（盯）+ Mode Manager                 | P1↔P2：DDS（QoS: deadline + liveliness） |

> 理由：Doer-Checker 双轨制衡需要进程级内存隔离；5 进程过度碎片化，Stage 1-3 串行通路同进程内零拷贝最优；Stage 4 需在 P1 崩溃时独立存活。

------

### F-02 内部通信总线 — **混合总线** （nlm-ask High）

| 路径                      | 机制                                     | 理由                                        |
| ------------------------- | ---------------------------------------- | ------------------------------------------- |
| P1 内部 Stage 1→2→3       | **lockfree SPSC ring buffer**（零拷贝）  | 1-5 Hz，同进程，最低延迟                    |
| P1 ↔ P2 监控              | **DDS Topic**（RTI Connext 或 Fast DDS） | QoS deadline/liveliness；CCS 追溯；工业标准 |
| Stage 4 紧急通道（100ms） | **共享内存 mmap + SCHED_FIFO**           | 绕过 DDS 序列化开销，保证 100ms 软实时      |
| ASDR 持久化               | **内部总线 → 加密日志**                  | 事件驱动，不影响主路径                      |

------

### F-03 算法插件加载 — **运行期动态多态** （nlm-ask High）

- **接口层**：纯虚基类（`IVesselDynamicsPlugin`、`IMPCPlugin`、`ICheckerPlugin` 等）定义 C++ 接口契约
- **加载层**：`dlopen()` + 工厂函数，启动时按船型/ODD 注入对应 `.so`
- **隔离保证**：骨架代码编译后不再需要重编译；各插件独立 V&V；CCS 验船师评估骨架白盒逻辑与插件接口契约

------

### F-04 Checker 流水线 — **差异化同步/异步** （nlm-ask High）

| Checker                       | 模式             | 延迟预算 | 理由                                                         |
| ----------------------------- | ---------------- | -------- | ------------------------------------------------------------ |
| **Checker-D**（决策合规校验） | **同步阻塞**     | < 100ms  | 5 条 STL 规则纯谓词逻辑，无仿真，100ms 内可完成；阻塞确保决策证书合规后才进 Stage 3 |
| **Checker-T**（轨迹安全校验） | **异步后台线程** | < 500ms  | 10-15 min 前向仿真，不可阻塞 MPC 主循环；Doer 输出提交队列，Checker-T 后台消费；连续 3 次失败 → MRC |

------

### F-05 实时调度 — **PREEMPT_RT + 硬件看门狗** （nlm-ask High）

| 组件             | 调度策略                  | RT 优先级  |
| ---------------- | ------------------------- | ---------- |
| Stage 4 紧急通道 | `SCHED_FIFO`，独立线程    | 90（最高） |
| Stage 4 常规监控 | `SCHED_FIFO`              | 70         |
| Stage 1-3 主决策 | `SCHED_RR`                | 50         |
| Checker-T 后台   | `SCHED_OTHER`（非 RT）    | —          |
| **硬件看门狗**   | 独立 MCU/FPGA，与 SW 解耦 | —          |

> 100ms 是**软实时目标**；硬性安全底线靠硬件看门狗（P1+P2 均崩溃时直接下发 emergency full-astern）。

## L3 架构评审报告

### 问题一：架构是否合理，是否当前最优？

**结论：合理，但不是最优。**

🟢 **候选 C 是"最佳安全壳架构（Best-in-Class Safety Architecture）"**，满足 CCS 入级和近期工业部署要求。

🟡 **但不是"最优认知架构"**，2020 年后有 3 个新范式已超越它：

| 新范式                                                  | 核心创新                                                   | 候选 C 的对应缺口                                            |
| ------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| **World Model-based RL** (VAE + Transformer, 2025 发表) | 隐空间内预测未来状态，"脑海演练"轨迹                       | Stage 1 运动预测器仅线性外推 + GP，无 latent dynamics；Stage 2/3 是反应式，不能提前"想象"3 艘船的意图演化 |
| **Neuro-Symbolic AI / Digital Captain** (DCAA 架构)     | 神经网络感知 + 符号推理 COLREGs，可向 ROC 解释"为什么右转" | 候选 C 的 COLREGs 是硬编码规则链，决策透明度低于符号层；当 ROC 远程监督时难以追因 |
| **SAVE**（Situation-Aware Verification）                | 运行时检测 ODD 漂移并动态更新安全属性阈值                  | 候选 C 的 RTA Simplex 是设计时固定的静态验证集，无法应对 ODD 外情景下的动态适应 |

另外，milliAmpere2（NTNU）采用了候选 C 没有的 **Control Barrier Functions（CBF）**：

| 安全机制    | 候选 C（RTA Simplex）                    | milliAmpere2（CBF）                |
| ----------- | ---------------------------------------- | ---------------------------------- |
| 约束方式    | 离散切换（正常→紧急一刀切）              | 连续推力约束（始终"推离"危险边界） |
| 过渡平滑性  | 存在跳变，Stage 4 紧急通道激活时行为突变 | 平滑渐进，无突然切换感             |
| 与 MPC 集成 | RTA 旁路在 MPC 之外                      | CBF 直接嵌入 MPC 代价函数          |

**架构质量综合评分**（相对 2026 年最前沿）：

- 安全性与合规性：🟢 优秀（CCS 入级充分）
- 认知能力（预测深度）：🟡 中等（反应式，缺 World Model）
- 可解释性：🟡 中等（规则链白盒，但无符号推理层）
- ODD 适应性：🔴 不足（静态 RTA，无 SAVE 循环）

用一句话说清楚：**候选 C 解决了"遇到船时怎么避"，但没有完全解决"航线中的所有战术决定"。**

| ---------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------------ |

### 最关键的两个结构性缺口

**缺口 1：L3 → L2 反向反馈通道缺失**

```
现状：L2 → L3（单向）：全局航路 + 时序约束
缺失：L3 → L2（反向）：战术层"请求重规划"信号
```

当前架构的隐含假设是 L3 的所有避让都在 Stage 3 的 5 分钟 MPC 时域内可解。一旦交通密度极高、TSS 限制、或 VTS 封航导致局部无解，唯一出路是 MRC（最小风险态）。这对 FCB 这类高度运营压力的船型过于保守。

**缺口 2：Stage 2 是"规则执行器"，缺少"情景类型路由器"**

完整战术层需要先判断"当前是什么战术情景"，再选择对应的决策子系统：

```
当前：所有情景 → Stage 2 COLREGs 6步推理链 → Stage 3 MPC
需要：情景路由 → [开阔水域会遇 | 狭水道/TSS | 进港 | 锚泊待泊 | 编队] → 对应子系统
```

没有情景路由器，就没有"进港模式"和"TSS穿越模式"——所有情况用同一套 COLREGs 推理链处理，在特殊场景下会给出错误的优先级判断。

### 对当前工作的建议

**短期（不改变 Spec v1.1 主体，补充缺口）**

1. **增加 L3→L2 上行反馈接口**：当连续 N 次 Doer-Checker 失败不是因为安全问题而是因为"全局航路与当前交通不兼容"时，触发 `route_replan_request` 上行到 L2，而非直接 MRC
2. **在 Stage 2 前增加情景路由器（Scenario Router）**：识别当前是开阔水域/TSS/狭水道/港口哪种情景，选择对应的 Stage 2 决策子路径
3. **补充 TSS 多边形约束到 Stage 3 Checker-T（CHK-06）**：Checker 增加一条 STL 规则：`always !(position inside TSS_separation_zone)`

**中期（v2 迭代时考虑）**

1. **CBF 嵌入 Stage 3 MPC 代价函数**：将 RTA Simplex 的硬切换替换为连续推力约束，平滑安全边界，减少紧急通道触发频率
2. **World Model 作为 Stage 1/2 的高级插件**：在复杂多船场景下启用，作为现有线性预测的升级替代

## L2 vs L3 功能切割

### 核心判断标准

NotebookLM 引用 Kongsberg K-MATE 和 Yara Birkeland 的实际分层方式，给出了清晰的切割原则：

| 维度         | L2（战略层 Voyage Planner）              | L3（战术决策层 TDCAL）                   |
| ------------ | ---------------------------------------- | ---------------------------------------- |
| **时间尺度** | 小时级、航次级                           | 秒级、分钟级                             |
| **处理对象** | 静态障碍（陆地、浅滩、TSS 车道边界）     | 动态障碍（他船、拖轮、渡轮、引航艇）     |
| **运行频率** | 离线或极低频（按需重算）                 | 1-5 Hz 持续运行                          |
| **输出形式** | 标称轨迹（Nominal Trajectory）+ 航次参数 | 实时决策修正（偏离标称轨迹后的避让动作） |
| **COLREGs**  | 不直接处理 Rule 13-19                    | 主要执行者                               |

### 进出港各阶段的归属

```
  港外 → 引航海域 → 港口航道 → 泊位前水域 → 泊位
  ─────────────────────────────────────────────────
  L2负责：
  ├── 选择进港航路（哪条引航道）
  ├── 计划 ETA + 申请引航
  ├── 向 VTS 提交 NOA（预计到港通知）
  ├── 锚地/泊位选择（具体编号）
  └── 生成进港标称轨迹（waypoint 序列）
  
  L3 负责（本次评审发现的缺口）：
  ├── B-02 TSS 车道内动态保持（Rule 10 路径规划感知）
  ├── B-03 VTS 实时指令执行（速度限制、等待许可）
  ├── B-04a 进港航道 COLAV（Rule 9 + 对向/交叉船流）
  ├── B-04b 引航员接送艇会合机动
  ├── B-04c VTS 等待指令时的 Holding Pattern 执行
  └── B-04d 拖轮会合定位（靠泊前最后 0.5nm）
  
  L4 负责（控制层，不在 L3 范围）：
  ├── 靠泊精确定位（DP 控制 / 横移推力分配）
  ├── 系缆作业协调
  └── 低速精确航向/速度控制（rudder + thruster allocation）
```

------

### 三层切割线的本质

```
L2 → L3 接口（已有）：
  "按这条标称轨迹前往泊位 X"
  → 输出：waypoint 序列 + 时序约束 + 优先级

L3 → L4 接口（已有）：
  "以这个航向/航速执行"
  → 输出：参考航向 + 参考航速 + 紧急级别

[缺失] L3 → L2 反向通道（B-01 问题的根源）：
  "当前进港航道严重拥堵 / VTS 要求等待 2 小时"
  → L3 需要能上报状态，触发 L2 重新计算 ETA 和到港顺序
```

**关键结论**：靠泊本身（Berthing）是 **L4 的精确控制问题**，不属于 L3。但进港过程中 **L4 开始前的所有战术机动**（引航员接送、等待 VTS 许可、拖轮定位）全部是 **L3 的职责**。

------

### 对 L3 Spec v2.0 的具体影响

这个切割结论告诉我们，B-04（进出港）不是一个"要不要加"的问题，而是需要在 **Scenario Router（A-05）** 里增加以下明确场景分支：

| Scenario Router 分支      | 触发条件                                    | 激活的 L3 子决策路径                                         |
| ------------------------- | ------------------------------------------- | ------------------------------------------------------------ |
| `scenario_open_sea`       | water_zone = open_sea                       | Stage 2 全 COLREGs 规则链（现有设计）                        |
| `scenario_coastal_tss`    | water_zone = coastal AND in_tss = true      | Stage 2 优先 Rule 10 + TSS 多边形约束（B-02）                |
| `scenario_narrow_channel` | water_zone = narrow_channel                 | Stage 2 Rule 9 优先（现有，需显式化）                        |
| `scenario_port_approach`  | water_zone = port AND speed < arrival_speed | Stage 2 切换进港模式：VTS 合规 + 引航员会合 + Holding Pattern（B-03/B-04） |
| `scenario_pilot_transfer` | pilot_requested = true                      | Stage 3 生成会合轨迹（特殊低速 MPC 参数）                    |
| `scenario_waiting`        | vtc_clearance_pending = true                | Stage 3 执行 Holding Pattern（固定点 DP 或圆形等待圈）       |

锚泊（B-05）则是 L2 触发、L3 执行进锚标称轨迹的 **L2→L3 场景**，L3 只需新增 `scenario_anchor_approach` 分支，处理接近锚地时的 COLAV，锚位选择本身由 L2 决定。
