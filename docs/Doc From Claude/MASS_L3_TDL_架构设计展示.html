C1 — CCS 合规约束

C2 — IMO MASS Code
能识别并响应包络越界
符合 MSC 110/111 MASS Code 草案要求：系统必须独立识别是否处于 Operational Envelope 之外，并触发规范化降级或 MRC 流程。
C3 — 多船型兼容
一套核心代码，无限船型
Capability Manifest + 水动力插件机制将船型参数完全外化。换船型时零修改决策算法，仅替换 YAML 配置文件和动力学 .so 库。
C4 — 人机工程
≥60s 接管时窗，SAT 三层透明
基于 Veitch et al. (2024) 实证研究，ROC 操作员接管时窗须 ≥ 60s。SAT-1/2/3 三层透明接口确保操作员始终知晓系统在做什么、为什么这么做、接下来会怎样。
C5 — 功能安全
IEC 61508 SIL 2 + ISO 21448 SOTIF
核心安全链路（ODD Monitor + MRC 触发）满足 SIL 2；ISO 21448 SOTIF 处理"系统按设计运行但设计本身不足"的场景——如 AIS 关闭目标的碰撞风险。
设计边界
TDL 范围：从感知输出到 L2 指令
TDL 上接 L2 战略层（Voyage Planner），下接 L4/L5 执行控制（Heading/Speed Controller）。不包含传感器融合内部算法、全局航次优化、舵机 PID。
Section 02
四项核心架构决策
以下四项决策驱动了整个 TDL 的架构形态。每项决策都有明确的被弃用替代方案和弃用理由。点击展开查看详细分析。
D1
ODD 作为组织原则，而非监控模块
ODD/Envelope Manager (M1) 是所有模块的中央调度枢纽。ODD 状态变化是驱动系统行为切换的唯一权威来源。
›
决策理由
IMO MASS Code MSC 110（2025）§15 要求"系统必须能独立识别是否处于 Operational Envelope 之外"。Rødseth et al.（2022）证明：海事 ODD 必须包含人机责任分配维度，不同于汽车行业 SAE J3016 的 ODD 定义。当 ODD 变化时，所有模块的行为语义同时改变——不是参数微调，而是语义切换。
被弃用的替代方案
旁路监控模式：算法先执行，ODD 偶尔检查。
✕ 弃用理由：各算法模块各自维护世界假设，当假设失效时没有统一更新机制，产生"部分模块认为安全、部分已越界"的一致性危机，是难以调试的隐性故障模式。
D2
8 模块功能分解（三层职责架构）
按包络控制层 / 决策规划层 / 安全接口层三层组织 8 个功能边界清晰的模块，对 DNV-CG-0264 9 个导航子功能实现 100% 覆盖。
›
决策理由
DNV-CG-0264 §4 将导航功能分为 9 个独立子功能（§4.2–§4.10），每个须独立验证。CCS 接受 DNV 等效性验证，因此本设计 8 模块对 9 子功能的 100% 覆盖是认证入口的必要条件。CMU Boss（DARPA Urban Challenge 冠军）的三层规划栈是按职责而非算法划分模块的最强工业证据。
被弃用的替代方案
候选 A（7 模块算法导向）：按算法功能切分，ODD 作为旁路监控，HMI 作为接口适配。
✕ 弃用理由：COLREGs 推理嵌入 Behavior Arbiter 无法独立 SIL 验证；HMI 多点分散无法保证责任移交原子性；无显式的 SOTIF 监督模块。
D3
CMM 映射为 SAT-1/2/3 三层透明性接口
船长心智模型（CMM）不作为系统内部状态机，而是通过三层 API 对外可观测：current_state / rationale / forecast+uncertainty。
›
决策理由
Chen et al.（ARL-TR-7180）的 SAT 模型实证：SAT-1+2+3 全部可见时操作员 SA 提升且工作负荷不增加。Veitch et al.（2024）实证证明 DSS 可用性是 ROC 绩效最强影响因子。M8 作为唯一 ROC/船长接口，确保"谁在控制"的答案在任何时刻唯一确定。
被弃用的替代方案
将 CMM 实现为"模拟船长决策"的内部状态机，外部暴露简单操作界面。
✕ 弃用理由：产生"算法语言 vs 船员语言"语义鸿沟——CCS 验船师、船长、ROC 操作员无法对同一场景达成共识。认证时无法向验船师解释"系统为什么做了这个决策"。
D4
Doer-Checker 双轨安全模式
M1–M6 作为 Doer（自主决策执行），M7 作为 Checker（运行时安全仲裁），Checker 逻辑须比 Doer 简单 100 倍以上且独立实现。
›
决策理由
Jackson et al.（MIT CSAIL, arXiv:2104.06178, 2021）从形式验证角度论证：主控制器找高性能解，安全监督器只验证解是否满足安全约束，后者的规约必须足够简单以支持形式化证明。Boeing 777 飞行控制系统和 NASA F/A-18 Auto GCAS 是 SIL 3 认证的同类工业先例。
核心设计要求
M7 的 6 类硬约束须能在 10ms 内完成检查（独立于 MPC 优化求解）：CPA 违反、TCPA 越界、速度超限、转艏率超限、ODD 边界越界、通信链路失效。

M7 有权直接向 M5 注入安全轨迹（绕过 M4 行为仲裁）——这是系统中唯一一处紧急快捷路径。
Section 03
ODD / Operational Envelope 框架
Operational Envelope = E × T × H（环境 × 任务 × 人机责任）三轴状态空间。这是 Rødseth et al.（2022）对汽车行业 ODD 概念在海事领域的关键扩展——H 轴（人机责任分配）是 MASS 独有的安全维度。
E — Environment（环境维度）
水域类型：开阔 / 狭水道 / 港内
气象能见度：≥2nm / <2nm (Rule 19)
海况 Hs：≤1.5m / ≤2.5m / >2.5m
交通密度：稀疏 / 中等 / 密集
水深裕量：UKC 百分比
T — Task（任务维度）
任务阶段：Transit / Approach / Berthing
速度域：巡航 / 低速 / DP精控 (<2kn)
作业类型：Crew Transfer / Standby / Port
VTS 区要求：是 / 否
锚地操作：是 / 否
H — Human Availability（人机责任）
控制位置：船上 / ROC / 无人
TMR：最大操作员响应时间
TDL：自主系统安全维持时间
通信链路质量：RTT / 丢包率
约束：TDL > TMR（安全约束）

Section 06
多船型参数化设计
三层解耦架构实现"一套核心代码，无限船型扩展"。换船型时 A 层（通用决策核心）零修改，仅更新 B 层（Capability Manifest）和 C 层（船型适配器）。
A 层 — Vessel-Agnostic Core（通用决策核心）
8 个 TDL 模块，无任何船型常量
所有决策算法从 Capability Manifest 读取参数，不硬编码任何船型特定值。换船型时本层代码零修改。
M1 ODD Manager M2 World Model M3–M6 决策模块 M7 Safety M8 HMI
B 层 — Capability Manifest（船型声明文件）
vessel_capability.yaml — CCS 签名验证
定义船型的所有参数化约束：主尺度、操纵极限（ROT_max、旋回半径、制动距离）、ODD 模板（各子域速度上限、CPA 参数）、推进配置、传感器配置、水动力插件路径。
loa / beam / draft ROT_max(speed) ODD templates hydro plugin path
C 层 — Vessel-Specific Adapter（船型适配器）
PVA 接口适配 + 水动力插件 + 传感器适配
标准 PVA 接口（Position-Velocity-Actuator）对外暴露 (ψ_cmd, u_cmd, ROT_cmd)；内部实现船型特定的 MMG / Abkowitz / 神经网络水动力模型。
PVA Interface MMG 4-DOF Plugin Sensor Adapter
Backseat Driver 范式
本设计直接继承 MIT MOOS-IvP 的 Backseat Driver 范式（Benjamin et al. 2010）：自主系统向船舶控制系统提供 (heading, speed) 指令，控制系统执行并回传导航信息。自主系统对控制系统的内部实现完全不感知。

Sea Machines SM300（TALOS + SMLink API）将此范式商业化，已在 200+ 艘船（20 国）部署——是当前最强工业验证。
